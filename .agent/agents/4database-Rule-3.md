# 🛡️ Database Rule: VPN Infrastructure & IPAM

**Target Agent:** `database-architect`
**Context:** Mikrogestor SaaS - WireGuard Tunnel Management
**Critical Goal:** Manage VPN Servers, allocate unique Internal IPs, and track Tunnel Lifecycle.

## 1. Infrastructure Topology (`public` Schema)

We need to treat VPN Servers as "Resources" with limited capacity.

### Table: `vpn_servers`
Represents a Docker container running WireGuard.
-   `id` (UUID): PK.
-   `name` (VARCHAR): e.g., "Gateway-BR-01".
-   `public_endpoint` (INET): The Public IP the MikroTik connects to.
-   `listen_port` (INT): e.g., 51820.
-   `public_key` (TEXT): Server's WireGuard Public Key.
-   `capacity_limit` (INT): Max tunnels (e.g., 250).
-   `current_load` (INT): Calculated count of active tunnels.
-   `is_active` (BOOLEAN): To drain connections if needed.

## 2. Tunnel Registry & IP Allocation

This is the most critical table. It links a Tenant (ISP) to a VPN Server and assigns an internal route.

### Table: `vpn_tunnels`
-   `id` (UUID): PK.
-   `tenant_id` (UUID): FK -> `tenants.id` (One tunnel per tenant usually).
-   `server_id` (UUID): FK -> `vpn_servers.id`.
-   `router_identity` (VARCHAR): e.g., "MikroTik-Main-CCR".
-   **IPAM Columns (Critical):**
    -   `internal_ip` (INET): The IP assigned inside the tunnel (e.g., `10.8.0.5/32`).
    -   `subnet_cidr` (CIDR): The logical network (e.g., `10.8.0.0/24`).
-   **Crypto Columns:**
    -   `client_public_key` (TEXT): Generated by Backend.
    -   `client_private_key` (TEXT): Encrypted (using pgcrypto or App encryption).
    -   `preshared_key` (TEXT): Optional security layer.
-   **Status:**
    -   `last_handshake` (TIMESTAMPTZ).
    -   `config_script` (TEXT): The full `/interface wireguard` script generated for the user.

### Constraints (Safety First)
1.  **Unique IP per Server:**
    `CONSTRAINT uq_server_ip UNIQUE (server_id, internal_ip)`
    *Prevents two ISPs from getting 10.8.0.5 on the same gateway.*
2.  **One Tunnel per Tenant (Optional):**
    `CONSTRAINT uq_tenant_tunnel UNIQUE (tenant_id)`

## 3. IP Allocation Logic (Function)

Don't rely on the application to "guess" the next IP. Create a Database Function.

```sql
CREATE OR REPLACE FUNCTION get_next_vpn_ip(p_server_id UUID)
RETURNS INET AS $$
DECLARE
    v_base_subnet CIDR := '10.8.0.0/16'; -- Example global subnet
    v_last_ip INET;
    v_new_ip INET;
BEGIN
    -- Find the highest IP currently allocated on this server
    SELECT MAX(internal_ip) INTO v_last_ip
    FROM vpn_tunnels
    WHERE server_id = p_server_id;

    IF v_last_ip IS NULL THEN
        RETURN '10.8.0.2'; -- Start after Gateway (.1)
    ELSE
        RETURN v_last_ip + 1; -- Postgres generic logic (needs cast)
    END IF;
END;
$$ LANGUAGE plpgsql;
